## 2. 产品背景与目标

### 2.1 产品背景

山东新高考志愿填报以“学校 + 招生专业（专业类）”为核心选择单位。考生需要同时考虑：

- 高考分数及全省位次；
- 选考科目要求；
- 当年招生计划；
- 历年专业最低分和最低位次；
- 院校、地区、专业、学费等个人偏好；
- 招生类型、校区、中外合作、校企合作等差异；
- 志愿顺序及冲、稳、保结构。

现有信息通常分散于招生计划、院校官网、一分一段表及历年录取数据中，普通家庭难以高效完成数据匹配和志愿整理。

本产品通过统一数据、选科过滤和算法预测，为考生提供专业级志愿筛选与志愿表编排工具。

### 2.2 产品目标

MVP 需要实现以下完整闭环：

```text
填写考生信息
→ 自动匹配位次
→ 过滤不符合选科要求的招生计划
→ 查看院校及专业录取数据
→ 查看录取概率和冲稳保标签
→ 加入志愿表
→ 调整志愿顺序
→ 检查志愿风险
→ 自动保存并导出 Excel
```

### 2.3 产品原则

1. **专业计划是最小业务单位**

推荐、收藏、加入志愿表、概率计算均基于：

```text
学校 + 招生专业（专业类）+ 招生类型 + 校区 + 招生年度
```

系统内部以唯一的 `plan_id` 识别一条招生计划。

2. **院校仅作为专业计划的分组展示维度**

不能仅将“某所学校”加入志愿表，也不能使用院校整体录取概率代替专业计划录取概率。

3. **位次优先于分数**

跨年度录取比较主要使用位次。分数用于匹配当年位次和辅助展示，不直接用不同年份分数进行简单比较。

4. **算法结果只作为辅助参考**

前端不自行计算录取概率及冲稳保标签。页面必须显示风险提示：

> 录取概率基于历史数据、招生计划及预测模型估算，仅供志愿决策参考，不代表实际录取结果或录取承诺。

5. **所有关键数据可追溯**

推荐页、详情页和导出文件应展示数据年度、更新时间或数据版本，避免用户误用过期数据。

## 3. 用户与使用场景

### 3.1 用户角色

#### 考生或家长

主要行为：

- 填写考生信息；
- 筛选学校和专业；
- 查看录取概率及历史数据；
- 创建和整理志愿表；
- 检查志愿表风险；
- 导出志愿方案。

#### 数据运营人员

主要行为：

- 上传一分一段表；
- 上传当年招生计划；
- 上传历年录取数据；
- 维护院校和专业外链；
- 处理数据校验错误；
- 发布数据版本。

#### 系统管理员

主要行为：

- 管理后台账号及权限；
- 查看数据导入记录；
- 查看系统操作日志；
- 管理年度基础配置。

### 3.2 MVP 账号规则

- 用户首次浏览可直接填写考生信息并查看推荐结果。
- 用户首次创建志愿表时，要求通过手机号验证码登录。
- 一个账号在同一高考年度默认维护一份考生档案。
- 一个账号可以创建多份志愿表。
- 不同年度的考生档案和志愿表相互隔离。
- MVP 暂不支持一个账号管理多名同年度考生。

## 4. MVP 范围

### 4.1 本期必须交付

1. 手机号登录及用户身份识别；
2. 考生分数、位次、选科及意向条件维护；
3. 根据一分一段表自动匹配位次；
4. 根据选科要求过滤招生计划；
5. 院校分组和专业计划展开；
6. 当年招生计划展示；
7. 近三年最低分、最低位次和招生人数展示；
8. 专业详情及院校详情；
9. 阳光高考、学校专业页面及招生官网外链；
10. 录取概率、预测位次区间及冲稳保标签；
11. 多份志愿表创建、复制、排序和删除；
12. 志愿表自动保存；
13. 志愿表基础检查；
14. 志愿表 Excel 导出；
15. 一分一段表、招生计划、历年录取数据 Excel 导入；
16. 数据校验、预览和发布；
17. 基础操作日志和数据版本管理。

### 4.2 暂不纳入 MVP

- AI 对话式志愿顾问；
- 自动生成完整的 96 个志愿方案；
- 基于职业测评或性格测评推荐专业；
- 院校或专业综合排名；
- 就业率、薪资、考研率等就业数据；
- 用户之间分享、评论或社区功能；
- 志愿表多人协作；
- App、小程序和原生客户端；
- 付费会员、订单及支付；
- 高考报名、志愿填报系统代操作；
- 自动抓取院校网站数据；
- 复杂的“院校优先”“专业优先”综合评分模型。

MVP 排序先提供可明确计算的客观字段排序，院校优先和专业优先综合排序在后续版本单独定义评分模型。

## 5. 核心业务对象与规则

### 5.1 招生计划唯一标识

每条当年招生计划生成唯一 `plan_id`，自然唯一键至少包含：

```text
year
school_code
major_code
enrollment_type
campus
education_level
```

即使学校和专业名称相同，只要招生类型或校区不同，也必须作为不同招生计划处理。

### 5.2 志愿重复判断

同一份志愿表中：

- 相同 `plan_id` 不允许重复加入；
- 相同学校和专业，但招生类型不同，允许同时加入；
- 相同学校和专业，但校区不同，允许同时加入；
- 不同志愿表之间允许加入同一条招生计划。

### 5.3 位次方向

位次数值越小，代表考生排名越靠前。

位次差定义为：

```text
位次差 = 预测最低位次 - 考生位次
```

- 位次差大于 0：考生位次优于预测最低位次；
- 位次差小于 0：考生位次落后于预测最低位次。

例如：

```text
考生位次：14,800
预测最低位次：16,200
位次差：1,400
```

对应说明为：

> 你的位次比预测最低位次靠前约 1,400 名。

原方案中“考生位次 23,000、预测位次 16,200、领先约 1,400 名”的样例存在数值矛盾，实际接口和测试数据必须遵守上述计算规则。

### 5.4 选科匹配

选科要求不能仅保存为展示文本，后台必须同步保存结构化匹配规则。

建议结构：

```json
{
  "display_text": "物理、化学",
  "match_type": "all",
  "subjects": ["物理", "化学"]
}
```

`match_type` 支持：

| 类型 | 含义 |
|---|---|
| all | 所列科目必须全部选择 |
| any | 所列科目选择任意一门即可 |
| none | 不限科目 |
| custom | 复杂规则，需人工配置或特殊解析 |

系统只能在结构化规则校验通过后发布招生计划。

### 5.5 数据状态

招生计划需要支持以下状态：

- 正常招生；
- 新增专业；
- 历史数据不足；
- 暂停招生；
- 已撤销；
- 数据待确认。

已加入志愿表的计划如果后续被暂停或撤销，系统不得自动删除，应在志愿表中标红并提示用户处理。

## 6. 核心用户流程

### 6.1 首次使用流程

```text
进入平台
→ 填写高考年份、分数和选科
→ 系统匹配位次
→ 补充报考层次和个人意向
→ 提交考生信息
→ 进入智能推荐页
```

### 6.2 推荐与加入志愿流程

```text
查看推荐结果
→ 使用条件筛选或关键词搜索
→ 展开院校专业列表
→ 查看专业详情
→ 点击加入志愿表
→ 登录或选择目标志愿表
→ 加入成功后继续浏览
```

### 6.3 志愿整理流程

```text
进入志愿表
→ 查看冲稳保数量
→ 拖动或输入序号调整顺序
→ 添加个人备注
→ 点击检查志愿表
→ 根据检查结果修改
→ 导出 Excel
```

## 7. 功能需求

### 7.1 登录与账号

#### 7.1.1 登录方式

MVP 使用手机号和短信验证码登录。

字段：

- 手机号；
- 短信验证码；
- 同意用户协议和隐私政策。

#### 7.1.2 登录触发时机

以下操作要求登录：

- 创建志愿表；
- 加入志愿表；
- 保存考生档案；
- 复制志愿表；
- 导出 Excel。

未登录用户可临时填写信息并浏览推荐结果。触发登录后，应保留当前页面状态，登录成功后继续原操作。

#### 7.1.3 验收标准

- 验证码错误时不能登录；
- 登录成功后恢复原筛选条件和浏览位置；
- 同一账号重新登录后可看到已保存的考生信息及志愿表；
- 用户退出登录后不能查看原账号志愿表。

---

### 7.2 考生信息页

#### 7.2.1 字段定义

| 字段 | 必填 | 规则 |
|---|---|---|
| 高考年份 | 是 | 默认当前开放年度，只能选择后台已发布年度 |
| 高考分数 | 是 | 整数，范围由年度配置决定 |
| 高考位次 | 是 | 默认自动匹配，允许手动修改 |
| 选考科目 | 是 | 从年度配置科目中选择，必须选择3门 |
| 报考层次 | 是 | 本科、专科、不限 |
| 意向地区 | 否 | 多选，未选择代表不限 |
| 意向专业 | 否 | 支持专业门类、专业类、具体专业 |
| 排除专业 | 否 | 支持专业门类、专业类、具体专业 |
| 学费上限 | 否 | 正整数，单位元/年 |
| 院校性质 | 否 | 公办、民办、不限 |
| 中外合作 | 否 | 接受、不接受、不限 |

#### 7.2.2 分数匹配位次

用户输入分数后，系统在对应年度的一分一段表中查找完全相同的 `score`，使用该分数对应的 `cumulative_count` 作为默认位次。

展示：

```text
系统匹配位次：23,000
数据来源：2026年山东省一分一段表
```

异常规则：

- 当前年度一分一段表未发布：提示用户手动输入位次；
- 输入分数不存在：提示检查分数，不自动取临近分数；
- 分数超出年度配置范围：不允许提交；
- 用户手动修改位次后，显示“手动填写”标识；
- 用户再次修改分数时，清除原手动位次并重新匹配。

#### 7.2.3 意向与排除优先级

- 意向专业用于推荐筛选和排序，不是硬性排除条件；
- 排除专业是硬性过滤条件；
- 同一专业同时出现在意向和排除列表时，以排除为准；
- “不接受中外合作”时，中外合作招生计划不进入推荐结果；
- 学费上限为空时不限制学费；
- 招生计划学费为空时不直接排除，但显示“学费待确认”。

#### 7.2.4 信息变更

修改以下字段后，应重新获取推荐结果并重新计算志愿表中的概率：

- 年份；
- 分数；
- 位次；
- 选考科目。

修改意向地区、意向专业、排除专业、学费、院校性质等字段后，只重新筛选结果，不必重新计算算法概率。

重新计算期间：

```text
录取概率重新计算中
```

原有志愿项不得丢失。

#### 7.2.5 验收标准

- 未选满3门科目不能提交；
- 位次必须为正整数；
- 修改选科后，不符合新选科要求的志愿项能被检查功能识别；
- 自动匹配位次和一分一段表数据一致；
- 手动位次有明确来源标识。

---

### 7.3 智能推荐页

#### 7.3.1 页面顶部信息

固定展示：

```text
分数：600
位次：约23,000
选科：物理、化学、生物
候选专业：1,286个
```

“候选专业”指经过以下硬条件过滤后的招生计划数量：

- 招生年度一致；
- 当前处于正常招生状态；
- 符合选科要求；
- 符合报考层次；
- 不属于用户排除专业；
- 符合“不接受中外合作”等硬性条件。

支持快速修改：

- 分数；
- 位次；
- 选科。

修改后弹窗提示：

> 修改分数、位次或选科后，推荐结果及已有志愿的录取概率将重新计算。

#### 7.3.2 MVP 筛选条件

| 筛选项 | 规则 |
|---|---|
| 冲、稳、保 | 多选，使用算法返回标签 |
| 录取概率 | 支持区间输入或区间滑块 |
| 院校名称 | 模糊搜索 |
| 专业名称 | 模糊搜索标准专业名及招生专业名 |
| 院校所在地 | 省份、城市多选 |
| 山东省内外 | 省内、省外、不限 |
| 报考层次 | 本科、专科 |
| 院校性质 | 公办、民办 |
| 院校标签 | 双一流等后台配置标签 |
| 专业门类 | 多选 |
| 专业类 | 多选 |
| 学费范围 | 最低和最高学费 |
| 中外合作 | 是、否 |
| 校企合作 | 是、否 |
| 当年招生人数 | 数值区间 |
| 计划变化 | 增加、减少、持平、新增 |
| 历史最低位次 | 以上一年最低位次为主进行区间筛选 |

筛选规则：

- 不同筛选项之间使用“且”；
- 同一筛选项内多选使用“或”；
- 支持“一键重置”；
- 刷新页面后保留当前筛选条件；
- 搜索关键词去除首尾空格；
- 无结果时展示可操作的空状态，不显示空白页面。

#### 7.3.3 MVP 排序方式

支持：

1. 录取概率从高到低、从低到高；
2. 位次差从大到小、从小到大；
3. 上一年最低位次；
4. 当年招生人数；
5. 学费从低到高、从高到低。

默认排序：

```text
录取概率从高到低
```

院校分组下的排序规则：

- 院校组按照该校符合条件专业中的最高排序值排列；
- 院校内专业按照所选排序字段排列；
- 排序值为空的专业排在有值专业之后。

“院校优先”和“专业优先”需要额外定义院校及专业评分模型，不纳入 MVP。

#### 7.3.4 院校分组卡片

展示字段：

```text
山东大学
山东·济南｜公办｜本科｜双一流
符合条件专业：18个
概率范围：42%—96%
```

交互：

- 点击院校名称进入院校详情页；
- 点击展开查看专业；
- 默认只展开一个院校；
- 切换筛选条件后保持页面滚动位置；
- 已展开院校内每次默认加载前20个专业，支持继续加载。

概率范围计算规则：

- 使用当前筛选结果中该院校专业的最低和最高概率；
- 算法未返回概率的专业不参与范围计算；
- 全部未返回时显示“概率待计算”。

#### 7.3.5 专业列表字段

| 字段 | 展示要求 |
|---|---|
| 专业名称 | 展示招生计划中的专业或专业类名称 |
| 招生类型 | 普通类、中外合作、校企合作等 |
| 校区 | 无校区数据时显示“以招生章程为准” |
| 选科要求 | 使用标准化展示文本 |
| 当年招生人数 | 例如12人 |
| 上一年招生人数 | 无数据时显示“—” |
| 计划变化 | 增加2人、减少3人、持平、新增 |
| 上一年最低分 | 无历史数据时显示“—” |
| 上一年最低位次 | 使用千位分隔 |
| 前两年最低位次 | 支持简要展示 |
| 学制 | 例如4年 |
| 学费 | 例如6,600元/年 |
| 录取概率 | 0—100的整数百分比 |
| 风险标签 | 冲、稳、保 |
| 数据状态 | 数据完整、新增专业、历史数据不足等 |

操作：

- 查看详情；
- 了解专业；
- 加入志愿表。

收藏功能不纳入 MVP，避免与志愿表功能重复。

#### 7.3.6 加入志愿表

点击“加入志愿表”后：

1. 未登录时先完成登录；
2. 用户没有志愿表时，默认创建“我的志愿表1”；
3. 有多份志愿表时，弹窗选择目标志愿表；
4. 默认加入目标志愿表末尾；
5. 已存在时提示“该招生专业已加入该志愿表”；
6. 加入成功后不跳转页面；
7. 按钮变为“已加入”；
8. 点击“已加入”可查看所在志愿表，但不直接删除。

达到志愿数量上限后：

> 当前志愿表已达到96项，请删除其他志愿或选择另一份志愿表。

志愿数量上限由年度配置决定，MVP 默认配置为96，不在前端硬编码。

#### 7.3.7 异常状态

算法服务不可用时：

- 正常展示招生计划和历史数据；
- 概率显示“待计算”；
- 标签显示“暂未评估”；
- 仍允许加入志愿表；
- 页面提示“录取概率暂时无法计算，请稍后刷新”。

历史数据不足时：

- 不伪造历史数据；
- 显示“历史数据不足”；
- 算法如仍返回结果，应同步展示置信度；
- 详情页说明预测依据有限。

#### 7.4.1 当前招生信息

展示：

- 学校名称；

- 标准专业名称；
- 招生代码；
- 招生类型；
- 所在校区；
- 教育层次；
- 当年招生人数；
- 选考科目要求；

- 学费；
- 招生备注；
- 单科要求；
- 外语语种要求；

招生备注较长时默认展示三行，支持展开全文。

#### 7.4.2 历年录取数据

| 年份 | 招生人数 | 最低分 | 最低位次 |
|---|---:|---:|---:|
| 2025 | 10 | 612 | 15,820 |
| 2024 | 10 | 608 | 16,430 |
| 2023 | 8 | 615 | 15,260 |

图表：

- 最低位次趋势折线图；
- 招生人数变化柱状图。

图表规则：

- 位次纵轴应明确说明“数值越小，排名越靠前”；
- 缺失年份不补零、不插值；
- 只有一年数据时不绘制趋势线，只展示数据卡片。

#### 7.4.3 概率信息

```text
录取概率：73%
标签：稳
预测最低位次：约16,200
预测区间：15,300—17,100
预测置信度：高
模型版本：v1.0
```

原因说明最多展示3条：

```text
你的位次比预测最低位次靠前约1,400名
今年招生计划增加2人
近三年最低位次波动较小
```

原因文本由算法服务返回，前端只进行格式化和敏感字符过滤，不自行生成结论。

算法返回的原因与数值明显冲突时：

- 记录接口异常日志；
- 隐藏冲突原因；
- 不影响其他概率字段展示。

#### 7.4.4 专业介绍外链

展示按钮：

```text
[阳光高考专业介绍]
[学校专业介绍]
[学校招生官网]
[学校官网]
```

跳转规则：

1. 优先使用精确专业页面；
2. 无学校专业页面时隐藏该按钮；
3. 无阳光高考具体专业链接时，跳转阳光高考专业搜索页；
4. 所有外链在新窗口打开；
5. 页面提示“外部网站内容由对应网站提供”。

数据字段：

```text
sunshine_major_url
school_major_url
school_admission_url
school_official_url
```

### 7.6 志愿表

#### 7.6.1 志愿表管理

用户可以：

- 创建多份志愿表；
- 修改志愿表名称；
- 复制志愿表；
- 删除志愿表；
- 查看最近保存时间；
- 设置当前使用的志愿表。

默认名称：

```text
我的志愿表1
我的志愿表2
```

限制：

- 单账号单年度最多创建10份志愿表；
- 名称长度1—30个字符；
- 志愿表名称允许重复；
- 删除志愿表需要二次确认；
- 账号至少保留一份志愿表。

#### 7.6.2 志愿表顶部统计

展示：

```text
已添加：56 / 96
冲：18个
稳：24个
保：14个
待评估：0个
```

统计以当前最新算法结果为准。

#### 7.6.3 志愿项字段

| 字段 | 说明 |
|---|---|
| 顺序 | 从1开始连续编号 |
| 标签 | 冲、稳、保、暂未评估 |
| 概率 | 算法返回值 |
| 学校 | 学校名称 |
| 专业 | 专业或专业类名称 |
| 招生类型 | 普通类、中外合作等 |
| 校区 | 具体校区 |
| 当年招生人数 | 当前发布数据 |
| 上一年最低位次 | 无数据时显示“—” |
| 学费 | 无数据时显示“待确认” |
| 数据状态 | 新增、停止招生、历史不足等 |
| 备注 | 用户个人备注 |

#### 7.6.4 顺序调整

支持两种方式：

##### 拖动排序

- PC 端拖动整行调整；
- 拖动过程中展示插入位置；
- 松开后立即更新顺序。

##### 输入序号移动

用户输入目标序号并确认：

- 当前志愿移动到目标位置；
- 中间志愿自动顺移；
- 输入小于1时移动到第一位；
- 输入超过当前数量时移动到最后一位。

排序完成后，所有序号必须连续且不能重复。

#### 7.6.5 删除

- 单条删除需要确认；
- 支持多选批量删除；
- 删除后后续序号自动前移；
- 删除操作保存失败时恢复原数据并提示用户。

#### 7.6.6 备注

- 每条志愿支持添加个人备注；
- 最多200个字符；
- 输入后自动保存；
- 备注不参与推荐和算法计算；
- 复制志愿表时同步复制备注。

#### 7.6.7 自动保存

以下操作触发自动保存：

- 添加；
- 删除；
- 排序；
- 修改备注；
- 修改志愿表名称。

页面状态：

```text
保存中…
已保存 14:32
保存失败，点击重试
```

技术及体验要求：

- 连续操作采用防抖保存；
- 用户关闭页面前如存在未提交内容，应立即执行保存；
- 服务端以最新操作版本号防止旧请求覆盖新请求；
- 保存失败不能错误显示“已保存”。

#### 7.6.8 考生信息变化后的处理

考生修改分数、位次或选科后：

- 志愿项保持原顺序；
- 概率和标签重新计算；
- 不符合新选科要求的志愿项标红；
- 已停止招生的计划标红；
- 页面显示“志愿表已根据最新考生信息重新评估”。

### 7.7 志愿表检查

#### 7.7.1 检查入口

志愿表顶部提供“检查志愿表”按钮。

点击后使用当前：

- 考生信息；
- 最新已发布招生计划；
- 用户偏好设置；
- 最新算法结果；

执行检查。

#### 7.7.2 检查级别

| 级别 | 含义 | 示例 |
|---|---|---|
| 错误 | 原则上不应继续保留 | 不符合选科、计划已撤销 |
| 警告 | 存在明显风险，需用户确认 | 缺少保底、超过学费上限 |
| 提示 | 信息不完整或需要关注 | 新增专业、历史数据不足 |

检查结果不强制禁止导出，但存在错误时，导出前必须再次确认。

#### 7.7.3 检查规则

##### 不符合选科要求

条件：考生选科不满足招生计划结构化选科规则。

级别：错误。

##### 重复招生计划

条件：同一志愿表存在相同 `plan_id`。

级别：错误。

正常前端操作不应产生重复项，该检查用于处理异常数据。

##### 计划停止招生或撤销

条件：志愿项对应计划状态不再是正常招生。

级别：错误。

##### 全部为冲刺专业

条件：

- 已评估志愿数量大于0；
- 所有已评估志愿标签均为“冲”。

级别：警告。

##### 缺少保底专业

条件：

- 已评估志愿数量大于0；
- “保”标签数量为0。

级别：警告。

MVP 不强制规定冲稳保数量比例。

##### 包含排除地区或专业

条件：志愿项命中考生档案中的排除地区或排除专业。

级别：警告。

##### 超过学费上限

条件：招生计划学费大于考生填写的学费上限。

级别：警告。

学费为空时显示“学费待确认”，不直接判定超限。

##### 中外合作或校企合作

条件：

- 用户选择“不接受”；
- 志愿项对应招生类型属于中外合作或校企合作。

级别：警告或错误，其中不符合明确硬性条件时按错误展示。

##### 新增专业或历史数据不足

条件：

- 无完整历史录取数据；
- 历史数据少于2个年度。

级别：提示。

##### 特殊要求

检查：

- 体检要求；
- 单科成绩要求；
- 外语语种要求；
- 性别等招生备注要求。

MVP 优先依赖后台结构化标签。无法结构化的招生备注，展示“请重点阅读招生备注”，不自动判定考生是否符合。

#### 7.7.4 检查结果交互

结果按“错误、警告、提示”排序。

示例：

```text
发现2项错误、3项警告、5项提示
```

点击检查结果：

- 自动滚动到对应志愿项；
- 对应行高亮；
- 展示具体原因和处理建议。

支持按问题类型筛选志愿项。

### 7.9 数据管理后台

#### 7.9.1 通用导入流程

```text
选择数据类型和年度
→ 上传 Excel
→ 系统解析
→ 字段校验
→ 数据预览
→ 查看错误
→ 修正或重新上传
→ 确认发布
```

导入数据不能直接覆盖线上正式数据，必须先进入待发布版本。

发布必须是原子操作：

- 全部成功后切换正式版本；
- 任意关键错误未解决时不能发布；
- 不能只发布部分有效行。

#### 7.9.2 导入批次状态

- 上传中；
- 解析中；
- 校验失败；
- 待发布；
- 已发布；
- 已取消。

每个批次记录：

- 批次编号；
- 数据类型；
- 年度；
- 文件名；
- 上传人；
- 上传时间；
- 总行数；
- 成功行数；
- 错误行数；
- 发布时间；
- 发布人。

#### 7.9.3 一分一段表

字段：

```text
year
score
same_score_count
cumulative_count
```

校验：

- 年度不能为空；
- 分数不能重复；
- 同分人数不能为负数；
- 累计人数必须随分数降低而单调增加；
- 累计人数不能小于同分人数；
- 分数范围必须符合年度配置。

#### 7.9.4 当年招生计划

核心字段：

```text
year
school_code
school_name
major_code
major_name
standard_major_code
major_category
campus
enrollment_type
education_level
subject_requirement_text
subject_requirement_rule
plan_count
tuition
duration
remark
plan_status
```

校验：

- 学校代码不能为空；
- 专业代码不能为空；
- 招生计划唯一键不能重复；
- 选科展示文本和结构化规则不能为空；
- 招生人数必须为正整数；
- 学费不能为负数；
- 学制格式必须合法；
- 招生类型和校区需要参与重复判断；
- 同名专业如果属于不同招生类型或校区，必须生成不同 `plan_id`。

#### 7.9.5 历年录取数据

建议字段：

```text
year
school_code
major_code
standard_major_code
school_name
major_name
enrollment_type
campus
plan_series_id
enrollment_count
min_score
min_rank
```

原方案的历史数据仅依赖学校代码和专业代码，可能将不同招生类型或校区的数据错误合并。因此 MVP 必须增加：

- `enrollment_type`；
- `campus`；
- 或稳定的 `plan_series_id`。

历史数据与当年计划应优先通过 `plan_series_id` 关联，不能只依赖专业名称。

校验：

- 最低分和最低位次不能为空；
- 招生人数不能为负数；
- 位次必须为正整数；
- 同年度、同计划序列不得重复；
- 无法匹配当年计划的数据进入待人工确认列表。

#### 7.9.6 院校和专业链接

维护字段：

```text
school_official_url
school_admission_url
sunshine_school_url
sunshine_major_url
school_major_url
```

匹配规则：

- 阳光高考专业链接按标准专业代码维护；
- 学校专业介绍按“学校 + 标准专业”维护；
- 院校官网及招生官网按学校维护；
- 允许批量导入和单条人工修改；
- 链接变更保留操作记录。

链接校验：

- URL 格式合法；
- 协议为 HTTPS 或 HTTP；
- 定期检测是否可访问；
- 外链不可访问不阻止核心招生数据发布，但需要提示运营处理。

#### 7.9.7 数据问题处理

错误数据支持导出错误明细 Excel，包含：

- 原始行号；
- 原始字段值；
- 错误字段；
- 错误原因；
- 修复建议。

## 8. 算法服务需求

### 8.1 职责边界

算法服务负责：

- 录取概率；
- 冲稳保标签；
- 预测最低位次；
- 预测位次区间；
- 置信度；
- 原因说明；
- 模型版本。

业务系统负责：

- 考生信息校验；
- 选科过滤；
- 招生计划和历史数据组织；
- 调用算法服务；
- 保存算法结果；
- 展示和异常降级。

前端不得自行推导冲稳保标签。

### 8.2 单条请求

```json
{
  "candidate": {
    "year": 2026,
    "score": 600,
    "rank": 14800,
    "subjects": ["物理", "化学", "生物"]
  },
  "plan": {
    "plan_id": 12345,
    "current_plan_count": 12,
    "history": [
      {
        "year": 2025,
        "plan_count": 10,
        "min_rank": 15820
      },
      {
        "year": 2024,
        "plan_count": 10,
        "min_rank": 16430
      },
      {
        "year": 2023,
        "plan_count": 8,
        "min_rank": 15260
      }
    ]
  }
}
```

返回：

```json
{
  "probability": 73,
  "label": "稳",
  "predicted_rank": 16200,
  "rank_range": [15300, 17100],
  "confidence": "高",
  "reasons": [
    "考生位次领先预测线约1400名",
    "招生计划增加2人",
    "历史位次波动较小"
  ],
  "model_version": "v1.0"
}
```

### 8.3 批量接口要求

推荐结果可能包含上千条招生计划，业务系统不能为每条计划单独发起一次网络请求。

MVP 必须由算法服务提供批量预测能力，或者由业务后端进行批量任务封装。

建议接口：

```text
POST /prediction/batch
```

单次支持最多200条招生计划，返回结果通过 `plan_id` 对应。

请求应携带：

- 考生信息；
- 招生计划数组；
- 数据版本。

返回应包含：

- 每条计划结果；
- 单条失败原因；
- 模型版本；
- 请求追踪编号。

### 8.4 缓存规则

算法结果缓存键至少包含：

```text
candidate_year
candidate_rank
candidate_subjects
plan_id
plan_data_version
model_version
```

用户仅修改意向地区、专业偏好或学费时，不重新调用算法。

### 8.5 异常与校验

- 概率必须为0—100的整数；
- 预测区间下限不得大于上限；
- 标签只能是冲、稳、保；
- `predicted_rank` 必须为正整数；
- 原因说明与返回数值冲突时记录异常；
- 单条失败不能导致整批推荐结果失败；
- 算法超时时业务页面按“待计算”降级。

## 10. 状态与异常处理

### 10.1 推荐页空状态

无符合条件结果时展示：

> 暂未找到符合当前条件的招生专业。

同时提供：

- 清除筛选条件；
- 放宽意向地区；
- 取消专业限制；
- 修改学费范围；
- 修改报考层次。

不能建议用户修改真实分数或位次。

### 10.2 数据缺失

统一展示规则：

| 数据 | 展示 |
|---|---|
| 学费缺失 | 待确认 |
| 校区缺失 | 以招生章程为准 |
| 历史分数缺失 | — |
| 历史位次缺失 | — |
| 概率缺失 | 待计算 |
| 外链缺失 | 隐藏对应按钮 |

### 10.3 数据更新

正式数据版本更新后：

- 推荐页使用最新版本；
- 志愿表不删除原志愿；
- 对发生变化的招生人数、学费、状态显示“数据已更新”；
- 用户打开志愿表时自动重新检查；
- 导出文件记录使用的数据版本。

## 11. 非功能需求

### 11.1 性能

建议 MVP 验收基线：

- 推荐页首次结果返回 P95 不超过3秒；
- 常规筛选和排序 P95 不超过2秒；
- 院校专业展开 P95 不超过1.5秒；
- 志愿表保存 P95 不超过1秒；
- Excel 导出在96条志愿规模下不超过10秒；
- 算法不可用时，基础招生计划查询仍可正常使用。

### 11.2 可用性

- 核心服务月度可用性目标不低于99.5%；
- 数据发布失败不能影响上一正式版本；
- 关键接口具备超时、重试和降级机制；
- 保存失败必须向用户明确提示。

### 11.3 安全与隐私

- 手机号在数据库中加密或脱敏存储；
- 日志不得记录完整手机号和短信验证码；
- 后台采用角色权限控制；
- 数据发布、修改和删除操作写入审计日志；
- 导出文件不包含手机号；
- 用户只能访问自己的志愿表；
- 所有接口必须校验登录身份和资源归属。

### 11.4 兼容性

PC 端优先支持当前主流版本：

- Chrome；
- Edge；
- Safari。

移动浏览器应支持：

- 填写考生信息；
- 浏览推荐；
- 查看详情；
- 加入志愿；
- 输入序号排序；
- 导出文件。

移动端不强制实现复杂拖动排序。

### 11.5 无障碍及可读性

- 冲稳保不能仅依靠颜色区分，同时展示文字；
- 概率、分数、位次使用统一数字格式；
- 重要风险提示不得使用浅色小字；
- 位次图表明确标注位次方向。

## 13. 核心验收场景

### 13.1 完整主流程

给定：

- 已发布一分一段表；
- 已发布当年招生计划；
- 已发布近三年历史录取数据；
- 算法服务正常。

用户能够：

1. 输入分数；
2. 自动获得正确位次；
3. 选择3门选考科目；
4. 查看符合选科要求的招生专业；
5. 查看当年招生人数和近三年录取数据；
6. 查看概率、标签和预测区间；
7. 将专业加入志愿表；
8. 调整顺序并填写备注；
9. 执行志愿检查；
10. 导出内容正确的 Excel。

以上流程中不得出现数据丢失或必须人工刷新才能继续的问题。

### 13.2 选科验收

准备满足和不满足选科要求的招生计划：

- 推荐结果只能显示满足规则的计划；
- 选科修改后推荐结果立即更新；
- 已加入但不再满足选科要求的志愿必须保留并标记错误；
- 志愿检查能够准确定位该志愿。

### 13.3 招生类型和校区验收

准备以下数据：

```text
同一学校
同一专业名称
普通类
中外合作
两个不同校区
```

系统应生成不同 `plan_id`，允许分别加入志愿表，不得错误判定为重复。

### 13.4 算法异常验收

算法服务超时或单条返回失败时：

- 招生计划列表仍正常展示；
- 对应计划显示“待计算”；
- 用户仍可加入志愿表；
- 志愿表统计显示“待评估”数量；
- 页面不得展示伪造概率。

### 13.5 数据更新验收

已加入志愿表的计划被设置为停止招生后：

- 计划仍保留在志愿表中；
- 显示停止招生标识；
- 志愿检查判定为错误；
- 导出时明确标记；
- 系统不得静默删除。

### 13.6 自动保存验收

用户连续执行添加、排序和修改备注后：

- 服务端最终数据与页面最终状态一致；
- 刷新页面后顺序和备注不丢失；
- 较早的保存请求不能覆盖较新的操作；
- 保存失败时页面明确显示失败。

## 15. 主要风险与处理方案

### 15.1 历史数据无法准确关联

风险：

不同年度专业代码、专业名称、招生类型或校区可能发生变化，仅通过学校和专业名称匹配会造成错误。

处理：

- 建立稳定的 `plan_series_id`；
- 保存标准专业代码；
- 将招生类型和校区纳入匹配；
- 无法确定的记录进入人工审核。

### 15.2 选科要求解析错误

风险：

选科要求文本可能包含“必须同时选择”“任选一门”等复杂语义。

处理：

- 不使用简单关键词直接完成正式发布；
- 保存结构化规则；
- 无法自动解析的规则必须人工确认；
- 数据未确认前不得发布。

### 15.3 概率被用户理解为录取承诺

处理：

- 推荐页、详情页和导出文件均显示风险声明；
- 避免使用“必录”“一定能上”等表达；
- 历史数据不足时展示置信度及数据状态。

### 15.4 当年数据频繁变化

处理：

- 招生计划版本化；
- 发布新版本时重新检查已有志愿；
- 志愿表展示数据变更提示；
- 导出记录数据版本和导出时间。

### 15.5 推荐列表算法调用量过大

处理：

- 必须使用批量预测接口；
- 对相同考生条件和计划版本缓存结果；
- 算法失败时基础查询降级可用；
- 不允许前端逐条直接调用算法服务。

1. 核心主流程全部通过验收；

8. 算法异常时系统能够正常降级；
9. 数据导入必须经过校验、预览和发布；
10. 上一正式数据版本不会因新批次导入失败而受影响；
11. 用户数据权限隔离测试通过；
12. 页面及导出文件均包含录取概率风险声明。
